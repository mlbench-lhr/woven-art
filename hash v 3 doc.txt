1.Hash Usage In General
In NestPay, Hashing approach is used in order to authenticate users for Gate operations. In this
document, Hash Version 3 is described.
NestPay has three versions of Hash approach which are called as “Version 1”, “Version 2” and
“Version 3”. From these versions, in “Version 1”, data is hashed with “SHA-1” hash algorithm
and then encoded with base-64, while in “Version-2” and “Version-3” data is hashed with
“SHA-512” hash algorithm and then encoded with base-64.
NestPay decides to use which hash version based on posted “hashAlgorithm” parameter to the
Gate. If value of “hashAlgorithm” parameter is “ver1”, NestPay uses Version 1 Hashing
Approach for Gate operations, while “ver2” value indicates usage of Version 2 Hashing
Approach, and “ver3” value indicates usage of Version 3 Hashing Approach.
WARNING: Hash Version 1 will no longer be in use due to PCI regulations. If you are currently
using Version1, you are strongly urged to switch to Version3. Version3 is more flexible and secure
compared to Hash Version 2, all merchants are urged to switch to Version3.
After hash value is calculated, it will be sent to Nestpay with the parameter named as “hash”.

2. Hash Version 1 and 2 Overview
Version 1 and Version 2 include specific parameters that need to be included in the hash in a
specific order, using different hash algorithms.

2.1 Hash Version 1 - Obsolete
To generate hash in hash version 1, the parameters given below must be concatenated together
in the order given below to create the plaintext for hashing. Then plaintext must be hashed with
SHA-1 hash algorithm and encoded with base64.
plaintext = clientid + oid + amount + okurl + failurl + transaction type + instalment + rnd +
merchantsafeauthtype +merchantsafekey +callbackurl + storeKey
Hash = Base64(SHA1(plaintext))

3.Hash Version 3
3.1 Sending Request to the NestPay Gate using Hash Version 3
To generate the hash ver3 for client authentication, append all posted request parameter values
except storeKey in alphabetical order (A to Z) using pipeline “|” as the separator between
parameters. If a parameter is sent to Gate with an empty value, it still will be added to hash
data (For an empty value example, Instalment parameter value can be checked in below
example).
After all request parameters are added alphabetically using pipeline “|” as separator, storeKey
will be added to hash data again using pipeline “|”.
Note that in a case of using “|” character in a parameter, “\” character is used for escaping.
Therefore, if the “\” character is also used in a parameter as its own value, we use “\” character
before it and then append it to hash plain text. For better understanding, you can check the
example below :
Original Value : ORDER-256712jbs\j6b|
Value used for Hash Calculation : ORDER-256712jbs\\j6b\|

3.2 Controlling Hash Value for the Response of a Gate Request
using Hash Version 3
For the Merchant System, it is needed to control hash value in the response in order to
understand whether response is really sent from NestPay or not. For this reason, NestPay posts
hash value with “HASH” parameter with its Response.
Calculation of Hash is done with the same manner in Request. NestPay, appends all posted
parameters in Response in alphabetical order (A to Z) using “|” as separator and then adds the

Store Key at the end of the hash data using pipeline “|” as separator. Later on hashes this data
using SHA-512 algorithm and encodes hashed data with base-64.
Important Note II: Parameters named “encoding”, “countdown” and “hash” are ignored
during hash calculation.

3.2.1 Usage of Customized Gate Templates with Hash Version 3
If Merchant has prepared customized Gate Result Templates for the Response of Gate and uses
Hash Version 3, there should be no post parameter in template other than inserted post
parameters by NestPay which are replaced with “@@allhiddenparams@@” line in the template.
If in the customized template, there exists post parameters other than added post parameters
by NestPay, calculated Hash Value in the Merchant Side will be different from Hash Value
Calculated by NestPay.
In a situation like this, if Merchant does not want to remove its customized templates which can
be used for Hash Version 2, another customized template can be added to system with a naming
which ends with “_ver3.htm” (Ex. gateresultlogosuccess_ver3.htm).
WARNING: Keep in mind that while templates can be same for Hash Version 1 and Hash Version
2 but Hash Version 1 should NOT be used.

4.Hash Version 3 Sample Codes
Sample Codes for Hash Version 3 are given with the “SampleCodes.zip” along with this
document. In this zip file, codes implemented in 4 programming languages, such as .Net C#,
.Net VB, JSP & PHP, can be found.

These codes are composed of three main sections for all four programming languages, such
as:
1. Request Data Preparation Part
2. Calculation of Hash Values Sent to Gate During Request
3. Checking Gate Response Hash Value to understand whether response is really sent from
Gate or not.
